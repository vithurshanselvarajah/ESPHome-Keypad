# Contributing

Thank you for contributing to this project. This document covers everything you need to develop, test, and submit changes.

---

## Prerequisites

| Tool | Version | Purpose |
|---|---|---|
| Python | ≥ 3.12 | ESPHome CLI, test runner |
| ESPHome | 2026.4.2 | Config validation, firmware compile |
| git | any | Version control |
| pytest | any | Running unit tests |
| yamllint | any | Linting automation YAML |

Install development dependencies:

```bash
pip install "esphome==2026.4.2" pytest yamllint
```

---

## Local Setup

```bash
# 1. Clone the repository
git clone <repo-url>
cd esphome

# 2. Create your secrets file
cp secrets.yaml.example secrets.yaml
# Edit secrets.yaml — fill in real WiFi, API key, OTA password

# 3. Validate the config (no compile, fast)
esphome config keypad.yaml

# 4. Compile the firmware
esphome compile keypad.yaml

# 5. Flash to device (USB)
esphome run keypad.yaml
```

## Tests

Tests live in `tests/`. They run without ESPHome or hardware — just Python.

```bash
# Run all tests
pytest tests/ -v

# Run a specific test file
pytest tests/test_project_structure.py -v
```

### What the tests check

| Test | Description |
|---|---|
| `test_secrets_example_covers_all_refs` | Every `!secret` reference in ESPHome YAML has a matching key in `secrets.yaml.example` |
| `test_secrets_example_has_no_real_credentials` | `secrets.yaml.example` contains only placeholder values, not real credentials |
| `test_secrets_example_exists` | `secrets.yaml.example` is committed to the repo |
| `test_secrets_yaml_is_gitignored` | `secrets.yaml` is listed in `.gitignore` |
| `test_keypad_yaml_packages_present` | `keypad.yaml` includes all required package files |
| `test_all_included_package_files_exist` | Every `!include` path in `keypad.yaml` resolves to an existing file |
| `test_fingerprint_backup_header_exists` | `keypad/include/fingerprint_backup.h` is present |

### What is NOT unit-tested

- **ESPHome YAML schema** — validated by `esphome config keypad.yaml`
- **Firmware correctness** — validated by `esphome compile` + hardware testing
- **fingerprint_backup.h C++ logic** — validated implicitly by the compile step; UART hardware is required for integration testing

## CI / CD

### Continuous Integration (CI)

Every push and pull request runs the CI workflow (`.github/workflows/ci.yml`):

```
push / PR
  │
  ├─ test       Run pytest on tests/
  ├─ lint       yamllint on automations/
  ├─ validate   esphome config keypad.yaml  (fast schema check)
  └─ compile    esphome compile keypad.yaml (full build, needs validate to pass first)
               └─ uploads firmware.bin as a workflow artifact (retained 30 days)
```

The compile job uses a PlatformIO cache (`~/.platformio`) so subsequent runs are fast after the first toolchain download.

All four jobs must pass before a pull request can be merged.

### Release Workflow

Releases are driven by a **version file + commit message convention** — no manual tagging.

#### Step 1 — Bump `VERSION`

Edit the `VERSION` file at the repo root to the new version number (semver, no `v` prefix):

```
1.2.3
```

#### Step 2 — Commit with a `release:` subject

The workflow only fires when the commit message **starts with `release:`**. The commit body is analysed by the AI to write the changelog — include any breaking changes, migration steps, or notable callouts there.

```bash
git add VERSION
git commit -m "release: v1.2.3 — brief one-line title" \
           -m "Longer description of what changed in this release.

Breaking: removed AP fallback mode — delete ap_password from secrets.yaml.
Migration: set debug_mode to 0 in keypad.yaml before flashing."

git push origin main
```

#### What happens automatically

```
push to main (commit starts with "release:")
  │
  ├─ Read VERSION (validate semver; fail if tag already exists)
  ├─ Extract commit subject + body
  ├─ Find previous release tag (for diff range)
  ├─ Call GitHub Models GPT-4o to analyse:
  │     - Full diff since previous tag
  │     - All commits in range
  │     - Commit body (breaking changes, author notes)
  │   → writes /tmp/changelog.md
  │   Falls back to plain git-log if AI is unavailable
  ├─ esphome compile keypad.yaml
  ├─ git tag -a vX.Y.Z + git push origin vX.Y.Z
  └─ gh release create:
       - Title: "Release vX.Y.Z"
       - Body: AI-generated changelog
       - Asset: firmware-vX.Y.Z.bin
```

#### Changelog AI

The changelog is generated by `scripts/generate_changelog.py` using the **GitHub Models API** (`gpt-4o`). The `GITHUB_TOKEN` that Actions provides automatically is used — no extra secrets needed. If the AI call fails for any reason (rate limit, network error, etc.) the script falls back to a plain list of commit messages.

The AI is instructed to:
- Check the commit body first (author-supplied notes take precedence)
- Reference actual scripts, entities, and YAML files by name
- Highlight breaking changes prominently with migration steps
- Skip empty sections

> **Note:** The compiled firmware uses dummy CI secrets. Users must compile from source with their own `secrets.yaml` to get a flashable binary for their device.

### CI Secrets

CI uses dummy credentials generated by `tests/generate_test_secrets.py`. These are syntactically valid for ESPHome to compile against but do not represent real WiFi or HA credentials. The generated `secrets.yaml` is never committed (it is gitignored).

## Submitting Changes

### Branch naming

| Prefix | Use for |
|---|---|
| `feature/` | New functionality |
| `fix/` | Bug fixes |
| `docs/` | Documentation only |
| `refactor/` | Code cleanup with no behaviour change |

### Pull request checklist

- [ ] All CI jobs pass (green checkmarks on the PR)
- [ ] New `!secret` keys are added to `secrets.yaml.example` with placeholder values
- [ ] New ESPHome package files are added to `packages:` in `keypad.yaml`
- [ ] If a new `!include` file is added, it exists on disk (checked by tests)
- [ ] Documentation updated if behaviour changed (see `docs/`)
- [ ] New HA automations in `automations/` pass `yamllint`

## File Structure

```
keypad.yaml               — Entry point; all user-facing substitutions
secrets.yaml.example      — Template for secrets.yaml (commit this, not secrets.yaml)
keypad/
  board.yaml              — ESP32-S3 chip config, on_boot logic
  network.yaml            — WiFi, HA API actions, OTA
  fingerprint.yaml        — R503 sensor, scan triggers, aura LED feedback
  keypad.yaml             — Matrix keypad, PIN collector, LED feedback scripts
  status_light.yaml       — WS2811 LED strip, brightness entity and scripts
  debug.yaml              — Optional: web server, diagnostics (debug mode only)
  include/
    fingerprint_backup.h  — C++ header: raw Grow UART backup/restore
automations/              — Ready-to-use HA automation examples
  README.md               — Index of all automation files
  pin-access.yaml
  fingerprint-access.yaml
  led-feedback.yaml
  led-brightness.yaml
  fingerprint-backup.yaml
  fingerprint-enroll.yaml
  notifications.yaml
  offline-alert.yaml
docs/                     — Project documentation
tests/
  test_project_structure.py   — pytest unit tests
  generate_test_secrets.py    — CI helper: writes dummy secrets.yaml
.github/
  workflows/
    ci.yml                — CI workflow (lint, validate, compile)
    release.yml           — Release workflow (compile + GitHub release)
.yamllint.yml             — yamllint config for automations/
```

## YAML Style Guide

### ESPHome YAML (`keypad/`)

- 2-space indentation throughout
- One blank line between top-level keys
- Lambda code: single-line where readable; use `|` blocks for multi-line
- Comments: use `#` on its own line above the relevant block, not inline

### HA Automation YAML (`automations/`)

- 2-space indentation
- `alias:` and `id:` on every automation
- Use `input_text` helpers for values that may change — avoid hardcoded PINs or slot numbers
- Use `trigger.event.origin` to scope automations to a specific keypad
- Keep Jinja2 templates in `>` (folded) blocks when longer than ~60 characters

## Debugging

### Compile fails locally

```bash
# Check the raw error output
esphome compile keypad.yaml 2>&1 | tee build_out.txt
grep -i "error:" build_out.txt | head -30
```

### Test the config without hardware

```bash
# Schema validation only (very fast, no toolchain needed)
esphome config keypad.yaml
```

### Enable debug mode

Set `debug_mode: "1"` in `keypad.yaml` and uncomment the `debug:` package line. This enables:
- ESPHome web server (port 80)
- Standalone testing without Home Assistant
- Additional diagnostic sensors

See [debug-mode.md](debug-mode.md) for full details.
